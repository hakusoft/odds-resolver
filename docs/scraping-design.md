# スクレイピングとデータ収集の設計

README の設計思想（append-only / 確定した過去は静的化 / 描画はブラウザ）を土台に、
**具体的な取得元を定めた収集層**の設計をここに定める。README が「思想」なら、
本書は「その思想を現実のソースに落とすときの具体判断」を扱う。

- 対象は **まず地方競馬**。ほぼ毎日開催で、修正 → 検証のサイクルが速いため。
  方法が確立したら中央競馬（JRA）へ流用する（場コード体系を抽象化しておく）。
- 実行基盤は **Lambda + EventBridge**（README のアーキテクチャ想定どおり）。
- 本書のスコープは **取得層と夜間バッチまで**。高頻度読み込み（1分毎）はこの土台の
  上に載る枝葉であり、確立後に別途詰める。

## 取得元の前提（2026-07-24 調査時点）

地方競馬の対象サイトを取得元とする。取得の可否と制約を先に確定させる。

### robots.txt

```
User-agent: *
Crawl-Delay: 60
Disallow: /app/
Disallow: /pcs/prediction/detail/RACEID/
```

- **オッズページ（`/odds/...`）は Disallow されていない** → 取得対象として許容される。
- **`Crawl-Delay: 60`（全クローラ）** → これは本設計の最重要制約。**同一ホストへの
  リクエスト間隔は 60 秒以上空ける**。この一点が取得スケジュールの形を決める。

### 利用規約（取得元の規約）

- **第33条（禁止事項）**：システムの妨害・混乱を禁止。
  → 過度な並列・高頻度アクセスは規約違反になりうる。行儀の良い低頻度取得に徹する。
- **第32条（知的財産権）**：転載・再利用を禁止。
  → 取得したオッズの**公開範囲**に関わる論点。本設計では **公開/非公開の最終方針は未確定**
    として先送りする（収集の器の設計はこれと独立に進められる）。データを S3 に置くところ
    までは共通で、「誰に配信するか」は別レイヤーの決定とする。

> **設計上の絶対制約**：Crawl-Delay 60 秒を守る。妨害にならない低頻度取得に徹する。
> この 2 点は以降のすべての判断の前提。

### URL 構造

レースは 18 桁のレース ID で一意に識別される（日付 + 場・回・日・R のコード体系）。

| 用途 | URL パターン |
| --- | --- |
| その日の開催一覧 | トップページ / 開催日ページ |
| 出馬表 | `/race_card/list/RACEID/{raceId}` |
| 単複オッズ | `/odds/tanfuku/RACEID/{raceId}` |

`raceId` は先頭 8 桁が `YYYYMMDD`。残りが場コード・回次・日次・レース番号を表す。
**この体系を場コードで抽象化しておけば、中央競馬（別体系）への差し替えが取得層の
パラメータ変更で済む。**

## 全体の器（収集フロー）

README のアーキテクチャ図を、取得元 + 夜間バッチ先行の方針で具体化する。
**今日固めるのは下図の実線部分。点線（高頻度）は後日。**

```
┌─────────────────────────────────────────────────────────────┐
│ 夜間バッチ（1日1回・深夜）  ← 今日の設計対象                  │
│                                                             │
│  EventBridge (cron: 深夜)                                    │
│      │                                                       │
│      ▼                                                       │
│  Lambda: archive                                             │
│    1. その日に取得済みの当日データ（DynamoDB）を読む         │
│    2. 確定分を view/(指標つきJSON) と raw/(Parquet) へ変換    │
│    3. S3 へ書き出し（静的化）                                 │
│    4. index.json（目次）を更新                               │
│    5. 移送済み当日データは DynamoDB TTL で自動失効（削除不要） │
└─────────────────────────────────────────────────────────────┘

┌ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┐
  高頻度読み込み（1分毎）  ← 後日。土台確立後に載せる
                                                              
   EventBridge (rate: 1分) → Lambda: fetch                    
     ・当日の生きているレースの単複オッズを取得               
     ・DynamoDB へ append（時刻つきスナップショット）         
     ・Crawl-Delay 60 の制約下でレースをどう回すかが論点      
└ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┘
```

### なぜ夜間バッチを先に作るか

append-only 設計では「確定した過去を静的化する」処理が土台で、高頻度取得はその上に載る。
アーカイブ形式（view/raw の JSON/Parquet スキーマ）が定まらないまま高頻度データを溜めると、
形式変更のたびに溜めたデータの作り直しになる。**器（アーカイブ形式）を先に固める。**

当面、高頻度取得が無い間は、夜間バッチの入力を「その日 1 回だけ取得した最終オッズ」に
できる。これで **1日1回の取得 → 整理 → 静的化** という最小ループを、高頻度の複雑さ抜きで
検証できる。地方競馬が毎日開催なので、この最小ループを毎日回して形を固める。

## データの形と置き場

現行のフロント（`frontend/data/races/*.json` + `index.json`）が読める形を出力の
ゴールとする。**収集層の最終成果物は、いまサンプルで使っている JSON と同じスキーマ**。

### 当日ホット（DynamoDB）

| 項目 | 内容 |
| --- | --- |
| パーティションキー | `raceId`（18桁） |
| ソートキー | 取得時刻 `HH:MM`（スナップショット単位で append） |
| 属性 | 各馬のオッズ配列、頭数、取得メタ（取得元URL・取得時刻） |
| TTL | 翌日以降に自動失効（夜間バッチで S3 へ移送済みのため削除処理は書かない） |

高頻度取得が無い初期は、1日1回の取得で 1〜数スナップショットだけ入る。
高頻度を載せると同 `raceId` 配下に時刻ソートで積み上がる（append-only がそのまま効く）。

### アーカイブ（S3）

README のとおり **view/（表示用）と raw/（探索用）を分離**する。

- `view/races/{raceId}.json` … 指標計算済みの整形 JSON。フロントが直読みする。
  現行サンプルの `races/*.json` と同スキーマ（horses / snapshots）。
- `raw/date={YYYYMMDD}/...parquet` … 列指向。Athena での探索的分析用。表示経路には出さない。
- `view/index.json` … 目次。現行の `data/index.json` と同形式（フラットな races 配列）。

**絵（SVG）までは焼かない**（README の責務境界のとおり）。view は数値まで。

## 取得層の実装方針（Lambda: fetch）

高頻度版・夜間版で共通して使う取得ロジックの方針。**実装は明日以降**だが、設計として先に。

- **VPC 外 Lambda**（README のネットワーク方針）。送信元 IP 可変を利点として使い、
  NAT Gateway の固定費を避ける。
- **Crawl-Delay 60 の遵守**：同一ホストへの連続リクエストは 60 秒以上空ける。
  複数レースを取る場合も、ホスト単位のリクエスト間隔がこれを割らないよう直列化する。
- **1 リクエスト = 1 レースのオッズ 1 ページ**（`/odds/tanfuku/RACEID/{raceId}`）。
- **User-Agent を明示**し、正体を隠さない。行儀の良いクライアントとして振る舞う。
- **失敗時は当該スナップショットを欠測として扱う**。append-only なので後続に影響しない
  （欠測は「その時刻の事実が無い」だけ。埋めない）。
- パース対象は HTML。DOM 構造の変化に備え、**パーサはスキーマ検証を挟み、想定外の
  構造なら欠測にフォールバック**して落とさない。

### Crawl-Delay 60 と高頻度読み込みの緊張（後日の論点）

「1分に1度」の高頻度取得は、Crawl-Delay 60 とちょうど境界。**単一レースなら 60 秒間隔で
両立するが、同時進行の複数レースを各 1 分で取ると、ホストへのリクエスト間隔が 60 秒を割る。**
この解き方（対象レースを絞る／取得を時間割りする／締切前だけに集中させる 等）は
高頻度フェーズで詰める。今日の夜間バッチ（1日1回）はこの緊張と無縁。

## 中央競馬への流用

- 差し替えポイントは **場コード体系と URL パターンだけ**に閉じ込める。
- 取得層は「raceId を渡すと単複オッズを返す」インタフェースに抽象化し、
  地方（対象サイト）/ 中央（別ソース）を実装差し替えで切り替えられるようにする。
- view/raw のスキーマは共通。中央でも同じフロントがそのまま読める。

## 今日決めたこと / 先送りにしたこと

**決めた**：
- 取得元は地方競馬の対象サイト。robots.txt でオッズページは許容、Crawl-Delay 60 を絶対制約とする。
- 夜間バッチ（1日1回・整理・静的化・アーカイブ移送）を先に作る。高頻度は後。
- まず地方競馬。中央は場コード抽象化で後から流用。
- 実行基盤は Lambda + EventBridge。取得は VPC 外 Lambda。

**先送り**（本設計と独立に決められる）：
- 公開/非公開の最終方針（規約第32条に関わる）。S3 に置くまでは共通なので収集設計は進む。
- 高頻度読み込みの具体スケジュール（Crawl-Delay 60 との両立）。
- view の指標スキーマ詳細（支持率以外の歪み・急変指標）。可視化の深化フェーズで詰める。
